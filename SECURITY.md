# Security

## Reporting Vulnerabilities

Do **not** open a public GitHub issue for security vulnerabilities. Instead:

1. Email **avik.burdwan@gmail.com** with:
   - A description of the vulnerability
   - Steps to reproduce
   - The potential impact
2. Expect an acknowledgement within 48 hours.
3. A fix or mitigation will be coordinated before any public disclosure.

---

## Design Scope

This is a **local development platform**. It is not hardened for production or shared-network use.

### In scope (protections provided)

- Secrets are generated randomly and stored only in `.env.local` (git-ignored)
- Kubernetes secrets are injected via `make kube-secrets` — never stored in Helm values files or committed YAML
- Docker Compose services use `${VAR:?error}` syntax to fail loudly on missing secrets rather than silently using defaults
- The `dags/` directory is a git submodule; a hook blocks direct edits to prevent accidental commits to the wrong repo
- All `depends_on` entries use health conditions to prevent race conditions that could cause data corruption on startup

### Out of scope (not protected by this repo)

- Network-level isolation between services (no TLS, no mTLS)
- Authentication on Trino, Iceberg REST, or internal Airflow APIs
- Container privilege restrictions (no `securityContext`, no read-only root filesystem)
- Production secret management (no Vault, no Sealed Secrets, no external secrets operator)
- Image supply-chain verification (no digest pinning, no Sigstore)

## Secret Rotation

Secrets are generated by `make secrets` and stored in `.env.local`. To rotate:

```bash
# Rotate a specific secret (leaves all others unchanged)
make rotate KEYS=MY_SECRET_KEY

# After rotating, re-apply Kubernetes secrets if the cluster is running
make kube-secrets
```

There is no automated rotation schedule. It is the operator's responsibility to rotate secrets periodically and after any suspected compromise.

## Production Deployment Checklist

This platform is not production-ready as shipped. Before deploying to any shared or internet-facing environment, complete all items below.

### Secrets & credentials

- [ ] All secrets moved to an external secret manager (Vault, AWS Secrets Manager, or similar)
- [ ] Kubernetes Secrets managed by Sealed Secrets or External Secrets Operator
- [ ] No `.env.local` on production hosts — secrets injected at runtime
- [ ] Secret rotation policy defined and automated
- [ ] All image tags pinned to digests (`image@sha256:...`), not mutable tags

### Network security

- [ ] TLS enabled on all ingress endpoints
- [ ] mTLS or network-level encryption between internal services
- [ ] Kubernetes NetworkPolicies enforced (deny-all default, explicit allow-list)
- [ ] Docker network segmentation: separate networks per trust boundary
- [ ] Nginx security headers added (`X-Frame-Options`, `Content-Security-Policy`, `Strict-Transport-Security`)
- [ ] Rate limiting configured on all public-facing endpoints

### Access control

- [ ] Authentication enabled on Trino, Iceberg REST, and Airflow API
- [ ] Kubernetes RBAC audited — no `cluster-admin` for workload service accounts
- [ ] MinIO access policies scoped to least-privilege per bucket
- [ ] Airflow RBAC roles assigned; default admin password changed

### Container hardening

- [ ] `securityContext.runAsNonRoot: true` on all pods
- [ ] `readOnlyRootFilesystem: true` where feasible
- [ ] `allowPrivilegeEscalation: false` on all containers
- [ ] Resource limits set on every container (CPU and memory)
- [ ] Pod Security Standards enforced at namespace level (`restricted` profile)

### Observability & compliance

- [ ] Kubernetes audit logging enabled and forwarded to a SIEM
- [ ] Container image vulnerability scanning in CI (Trivy or Grype)
- [ ] Backup and restore tested for PostgreSQL and MinIO data volumes
- [ ] Incident response runbook documented
- [ ] Security review completed by a qualified reviewer before go-live
